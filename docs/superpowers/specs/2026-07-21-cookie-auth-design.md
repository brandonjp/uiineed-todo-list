# Cookie-session auth — replace HTTP Basic Auth

**Status:** approved 2026-07-21 · **Target:** todo.bpf.fyi (DreamHost shared, Apache + PHP 8.2)

## Problem

The site is gated by Apache HTTP Basic Auth (`.htaccess` + `.htpasswd`). On iPhone
this is painful and unfixable at the config level:

- iOS password managers (iCloud Keychain, 1Password) **cannot autofill HTTP Basic
  Auth dialogs** — an OS limitation, not a setting.
- Basic Auth is stateless: no cookie, no session, no adjustable lifetime. In
  home-screen PWA / standalone mode Safari re-negotiates and fires several `401`s
  before surfacing the credential dialog — the "tries multiple times, then slowly
  asks" delay.

## Reframe

The app shell (HTML/CSS/JS) is already **public on GitHub** — no secret lives in
it. The only server-side asset worth protecting is the **synced todo data**
(`sync.php`'s single JSON blob, stored outside the web root). So the goal is:
(1) lock the data, (2) put a login wall on the front door, (3) do it with a
mechanism iPhones handle well.

## Approach (chosen: form-based cookie session, self-hosted)

Replace Basic Auth with a normal HTML login form (which password managers DO
autofill) that sets a long-lived, HMAC-signed cookie. A tiny PHP guard protects
the front door and the data endpoint. Everything stays on DreamHost; no DNS
changes, no third parties.

Rejected alternatives: **Cloudflare Access** (requires moving nameservers +
email-OTP login, more taps on iPhone than a saved password) and **token-gates-
data-only** (leaves the shell fully public and balloons scope for device
revocation / temp devices).

## Secrets — location & handling

One file **outside the web root**, mirroring the existing `.htpasswd` and
`~/todo-sync` patterns:

    /home/todo_bpf_dh_4wc3y9/todo-auth/config.php   (chmod 600)

    <?php return [
      'password_hash' => '$2y$12$…',   // bcrypt hash of the passphrase — never plaintext
      'signing_key'   => '<64 hex chars>',  // HMAC key for the cookie (openssl rand -hex 32)
    ];

- **Never committed** — the repo is public. Committed code (`auth.php`) only
  *reads* this file and contains no secrets.
- Password is generated by Claude, hashed on the server via `password_hash()`
  (plaintext piped over stdin, never in argv/history), and the plaintext is
  handed to Brandon in `~/Desktop/todo.bpf.fyi-password.txt` (+ clipboard) to
  save into his password manager.
- Change password later: `ssh todo "php -r '…password_hash…'"` → paste new hash.

## Security posture

- **Cookie** = `base64(expiry) . HMAC_SHA256(expiry | password_hash, signing_key)`.
  Binding the signature to `password_hash` means a password change auto-invalidates
  every existing session. Flags: `HttpOnly`, `Secure`, `SameSite=Lax`.
- **Password** stored as bcrypt; verified with constant-time `password_verify`.
- **HTTPS** stays force-redirected in `.htaccess`; cookie is `Secure`-only.
- **Brute force:** single strong passphrase + a failed-attempt throttle
  (sleep on failure). One-person app → guessing is hopeless.
- **Attack surface:** only `login.php` (a form) and the front door (redirects to
  it) are public; `sync.php` returns `401` without a valid cookie. The guard is
  the first statement in every protected file — no unauthenticated path to data.
- **Static assets** (`/public/*`) remain publicly readable — identical to the code
  already on GitHub, zero secrets. Locking them adds complexity for no gain: skip.
- `robots.txt` + `noindex` header retained for crawler defence.

## Components (all new files committed, secret-free)

- **`auth.php`** — reads the outside-web-root config; `todo_is_authed()`,
  `todo_set_cookie($remember)`, `todo_clear_cookie()`, `require_auth()`
  (redirect browsers to `login.php`; `401 JSON` for `sync.php`).
- **`login.php`** — autofillable form (prefilled dummy username field for reliable
  manager association + a `current-password` field) and POST handler; "Remember me
  on this device" (checked = 1-year persistent cookie; unchecked = session cookie
  → the temp/friend's-device answer). Throttles failures.
- **`logout.php`** — clears the cookie, redirects to `login.php`.
- **`index.html` → `index.php`**, **`index-zh.html` → `index-zh.php`** — prepend
  `<?php require __DIR__.'/auth.php'; require_auth(); ?>`; fix the intra-site
  `index*.html` links/redirects to `.php`.
- **`sync.php`** — add `require __DIR__.'/auth.php'; require_auth();` at the top;
  drop the "relies on Basic Auth" note.
- **`.htaccess`** — keep force-HTTPS + `noindex` + deny-dotfiles; **remove** the
  `AuthType Basic` / `Require valid-user` block; add
  `DirectoryIndex index.php index.html` and a 301 from a stale `/index.html` to `/`.
  Then delete `.htpasswd` on the server.

## Verification

- `curl -I https://todo.bpf.fyi/` → `302` to `login.php` (no cookie).
- `curl -I https://todo.bpf.fyi/sync.php` → `401`.
- Log in via browser → land on app, data loads; relaunch home-screen app → no
  prompt, no delay; password manager offers to save/fill.
- `curl` with a forged/edited cookie → still `302/401`.
